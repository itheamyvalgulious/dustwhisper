# 游戏玩法计划书

## 当前状态

本文件当前已经固定两块内容:

- `LLM/SLM -> 实时施法 / 创造法术` 的玩法结构。
- 世界观中的首版 `物质构成 / 光构成 / 气体模型方向`。

以下内容暂不展开:

- 具体地形生成规则。
- 具体怪物阵容。
- 更细的魔法世界观细则。
- 各反应的精确数值与覆盖特例。

这些内容留到后续讨论后再补。

## 1. 总体玩法定位

当前游戏定位为 `探索 + 战斗`。

玩家在探索过程中读取环境, 判断敌人和地形, 然后通过语音表达施法意图。游戏的核心 skill 不是背固定按键连招, 而是:

- 选择正确的法术。
- 用正确的表达说明目的。
- 在正确的位置和强度释放。

玩家基础资源至少包括:

- `HP`, 生命值。
- `MP`, 蓝量。

## 2. 实时施法的核心约束

### 2.1 装备栏

- 玩家同时只装备 `6` 个法术。
- 实时施法阶段, 系统不现场创造新法术。
- 实时施法只允许在已装备的 `6` 个法术中选择其一并释放。

这样做的目的有 3 个:

- 把实时问题从“全空间生成”收束成“有限候选选择”。
- 让玩家构筑本身成为玩法。
- 让模型的延迟和稳定性可控。

### 2.2 Bullet Time

- 玩家进入施法状态时, 游戏进入 `bullet time`。
- `bullet time` 的作用不是暂停世界, 而是给玩家一个短暂的思考和表达窗口。
- 这一窗口内, 系统完成语音转写, 法术选择, 定位解析, 强度估计和释放预览。

`bullet time` 是必要机制, 因为本作施法输入不是纯按键, 而是自然语言 + 语音风格。

### 2.3 实时施法拆成 3 个独立子问题

实时施法不应由一个“万能模型”一次性完成, 而应拆成 3 个不同模块:

1. `法术选择`
2. `定位解析`
3. `强度估计`

这 3 件事在性质上不同:

- `法术选择` 是对玩家当前已装备法术的检索 / 排序问题。
- `定位解析` 是固定语义空间中的目标 / 相对位置提取问题。
- `强度估计` 更接近音频风格 / 情绪 / 抑扬的估计问题。

其中“释放方式”不属于实时 `SLM` 解析职责, 而应在法术创造阶段就被法术自身定义好。

## 3. 实时施法流程

建议的实时流程如下:

1. 玩家按住施法输入, 游戏进入 `bullet time`。
2. `STT` 把语音转为文本。
3. `法术选择器` 在当前装备的 `6` 个法术中做排序。
4. `定位解析器` 从文本中提取定位相关语义。
5. `强度估计器` 从原始音频中提取强度相关信号。
6. 游戏本地 `resolver` 根据解析结果, 当前局部环境, 以及被选法术预先定义的释放方式, 计算真正的落点 / 起点 / 作用区域。
7. 游戏给出一个极短的施法预览。
8. 玩家松开按键或确认后, 正式释放。

这里的关键原则是:

- 模型不直接输出世界坐标。
- 模型不直接改世界状态。
- 模型只输出结构化意图。
- 游戏代码负责把“意图”解引用成实际施法结果。

## 4. 法术选择模块

### 4.1 问题性质

`法术选择` 不是传统意义上的全量分类器。

因为法术是玩家自主创造的, 法术集合不是预先写死的, 所以实时选择问题应被定义为:

- 输入: 本次语音文本 + 当前装备的 6 个法术元数据。
- 输出: 当前 6 个法术的相关性排序, 以及最终选中的法术。

也就是说, 它本质上是 `6 选 1` 的 rerank 问题, 而不是“从固定法术全集中分类”。

### 4.2 前提条件

要让 `法术选择器` 稳定工作, 每个法术在被创造出来时, 就必须同步生成一份机器可读元数据。实时阶段选择器依赖这份元数据, 而不是只看法术名字和文案。

这份元数据至少要能支持:

- 判断这个法术通常适合解决什么问题。
- 判断玩家当前表达是否像是在调用它。
- 提供后续运行时执行所需的法术定义。

### 4.3 设计结论

- 实时阶段不训练“认识所有法术”的统一分类器。
- 实时阶段只做“读懂当前这句话更像在用哪一个已装备法术”。
- 法术创造阶段必须为后续的实时选择准备好机器可读特征。

## 5. 定位解析模块

### 5.1 这是当前最难的一块

`定位解析` 是实时施法中智力要求最高的一部分。

它的职责不是直接产出坐标, 而是从自然语言里抽出可供本地世界查询的定位语义。

这部分语义空间相对固定, 因而适合提前训练。

### 5.2 当前只确定到模块边界

目前只确定:

- `SLM` 负责从自然语言中提取目标和相对位置语义。
- 本地 `resolver` 负责在当前环境中把这段语义解成真正可释放的位置。
- 定位语义当前至少包括 `anchor_filters`, `direction`, `distance` 这三类信息。

目前已经明确的 `anchor_filters` 大致分为 3 类:

- 怪物类, 如 `enemy`, `flying`, `walking`, `static`, `big`, `small`。
- 方位类, 如 `nearest`, `forward`, `backward`。
- 地形类, 如 `wall`, `hole`, `pool`, `hill`, `tree`。

其中地形类的意义不是写死名字, 而是通过环境匹配得到。例如 `wall` 可以被理解为“高度超过若干 cell, 宽度小于若干 cell, 两侧是空地”的局部地形模式。

`direction` 当前先收束为:

- `left`
- `right`
- `up`
- `down`
- `forward`
- `backward`

`distance` 当前先收束为:

- 数值距离, 单位为 `m`, 再按固定比例换算成 cell。
- 描述型距离, 至少包括 `near` 和 `far`。

### 5.3 例子

如果玩家说:

`在那只怪上方一米处落一根冰枪`

则定位解析器应提取出类似“目标是谁, 相对它的什么位置, 大致多远”这样的语义信息, 然后由本地 `resolver` 去当前局部环境中寻找:

- “那只怪”在当前上下文里指谁。
- 其上方 `1m` 是否可释放。
- 若不可释放, 应如何回退到合法方案。

### 5.4 设计结论

- 模型只抽象语义, 不负责世界几何求解。
- 世界求解必须由本地代码完成。
- `定位解析` 应保持在固定语义空间内, 而不是自由文本直接到坐标。
- 其中 `anchor_filters` 是最重要的一层, 因为它负责把玩家的自然语言指向“哪一类目标或哪一类地形”。

## 6. 强度估计模块

### 6.1 不主要依赖文本

`强度` 不应主要从 `SLM` 文本结果中得到, 而更应依赖原始音频。

原因是:

- 强度和稳定度跟语气, 音量, 语速, 停顿, 抑扬更相关。
- 单靠转写后的文本, 会丢失大量声音信息。

### 6.2 建议拆成两个值

建议强度模块不要只输出一个“威力”, 而是至少输出:

- `potency`, 本次施法的强度倍率。
- `stability`, 本次施法的稳定度。

这样可以支持:

- 说得更有仪式感, `potency` 更高。
- 说得混乱, 情绪失控, 或表达和法术风格冲突时, `stability` 更低。
- 某些法术在低 `stability` 下偏移, 衰减, 甚至反噬。

### 6.3 当前设计目标

- `potency` 决定威力, 范围, 持续时间或生成规模的倍率。
- `stability` 决定偏移, 失败率, 反噬率或自损风险。

这样“辱骂导致法术攻击自己”不需要写成全局硬规则, 而可以表现为:

- 某些法术系和某类表达风格高度不兼容。
- 不兼容时, 本次施法的 `stability` 显著下降。

## 7. 释放方式的归属

`释放方式` 不属于实时 `SLM` 解析的一部分。

当前更合理的归属是:

- 在 `LLM` 创造法术时, 就把这个法术的释放方式定死。
- 实时阶段只负责选择法术, 解析定位, 估计强度。
- 本地 `resolver` 按该法术预先定义好的释放方式执行。

也就是说, 玩家实时输入影响的是“这次把法术放到哪里, 以多大强度放”, 而不是临时改变该法术本身的释放机制。

## 8. 实时施法预览

实时施法强烈建议有一个很短的“解析预览”。

例如:

```text
法术: 冰枪
目标: 前方最近飞行怪
落点: 目标上方 1m
强度: 中高
```

这个预览不是为了让玩家读长文本, 而是为了降低“模型没懂我”的挫败感。

首版建议:

- 玩家按住施法键进入 `bullet time`。
- 预览在极短时间内显示。
- 玩家松开施法键时确认释放。
- 若玩家发现解析明显错误, 可以中断。

## 9. 实时施法的失败与代价

实时施法不应该只存在“成功或什么都没发生”两种结果。

建议至少存在以下几类失败后果:

- 目标无效, 法术落到最近合法点。
- 强度不足, 法术弱化释放。
- 稳定度过低, 法术偏移。
- 稳定度极低, 法术反噬或自损。

蓝量消耗建议由以下因素共同决定:

- 法术基础耗蓝。
- 本次 `potency`。
- 法术自身预设的释放方式。
- 实际作用范围。
- 是否发生失稳补偿或强行校正。

## 10. LLM 创造法术流程

### 10.1 这是非实时的强模型流程

`LLM 创造法术` 不应属于帧级实时决策链路。

它更适合作为:

- 战斗外行为。
- 安全窗口中的长耗时行为。
- 成长和构筑层玩法。

它的目标不是替代实时施法, 而是给玩家创造新法术, 扩展自己的 6 格装备池。

### 10.2 创造法术时允许的产物

当前初步构想中, `LLM` 侧的创造能力应拆成 4 层:

1. `LLM` 创造反应。
2. `LLM` 创造物质 / 气体。
3. `LLM` 创造 `ForceSource`。
4. `LLM` 创造法术。

它们的关系是:

- 反应是最基础层。
- 物质 / 气体建立在反应层之上。
- `ForceSource` 是和法术并列的一层能力, 不是法术之下的附属描述。
- 法术是最高层玩法对象, 它调用前面已经存在的内容。

法术本身当前按 5 个流派理解:

- `Gas`
- `Material`
- `Light`
- `Force`
- `Change`

其中:

- `Gas`, `Material`, `Light` 对应法术载体。
- `Force` 对应创建或调用 `ForceSource`。
- `Change` 对应对一个区域做直接修改。

进一步说:

- `Material` 流派的核心是创建物质, 再由该物质自身携带的反应生效。
- `Gas` 流派的核心是创建气体, 再由该气体自身携带的反应生效。
- `Light` 流派不是创建新的光定义, 而是使用现有光作为载体, 绑定不同反应。

当前对 `Change` 的限制是:

- 只允许修改区域内物质类型。
- 只允许修改区域内温度。
- 只允许修改区域内速度。

除此之外, 暂不允许扩展更多区域修改类型。

当前仍暂不允许 `LLM` 创造新的光定义。也就是说, `Light` 可以作为法术流派存在, 但其变化主要来自“现有光 + 不同反应”的组合, 而不是发明新的光本体。

### 10.3 创造法术的输出不应直接是文案

`LLM` 创造法术时, 输出不应只有名字和描述, 而必须同时产出机器可读结构。

当前只确定两点:

- 创造阶段必须生成面向实时系统的压缩语义, 当前至少包括 `keywords`。
- 如有需要, 可以额外保留一个 `vector`, 用于运行时做余弦相似度匹配。
- 创造阶段必须生成可供引擎执行和可供实时施法系统引用的法术定义。

除此之外的具体字段暂不在本文件中定死。

### 10.4 创造法术必须经过编译与校验

`LLM` 不应直接把生成结果写进游戏规则层。

中间必须存在一个 `法术编译器 / 校验器`, 用于:

- 检查是否使用了当前版本支持的原语。
- 检查是否引用了非法或不存在的载体。
- 检查是否超出当前版本的复杂度预算。
- 归一化为稳定的 `effect_ir`。
- 自动估算耗蓝, 风险, 稀有度和其他平衡参数。

这里的设计原则是:

- `LLM` 负责提出设计。
- 编译器负责判断“这能不能在当前游戏里成为一个合法法术”。

同时, 法术的释放方式也应在这一阶段被确定下来, 而不是交给实时 `SLM` 再做一次决定。

### 10.5 关于新物质 / 新气体 / `ForceSource`

如果玩家创造的法术引入了新物质, 新气体或新的 `ForceSource`, 那么它不仅是在造一个法术, 也是在扩展玩法词汇表。

这类内容最终至少需要回答:

- 它是什么载体。
- 它如何参与已有反应系统。
- 它的基础危险度和资源代价是什么。
- 它在实时施法中如何被选择和释放。

但这些规则目前先不在本文件定死, 因为它直接依赖后续的“物质 / 魔法世界观”讨论。

## 11. 首版物质构成

### 11.1 当前已经确定的物质大类

当前世界中的首版物质构成先按以下几类理解:

- `土石类`
- `植物类`
- `金属类`
- `常规水系`
- `危险液体系`
- `特殊功能物质`

其中:

- `土石类` 是地形, 支撑, 崩塌和粉化的主要来源。
- `常规水系` 当前先只保留 `water` (`水`) 这一种主物质。
- `危险液体系` 当前明确包括 `poison` (`毒液`), `acid` (`酸液`), `oil` (`油`)。
- `特殊功能物质` 当前明确包括 `fire` (`火焰`), `explosive` (`炸药`), 多种 `phosphor_*` (`荧光粉末`), `pollution` (`污染物`), `vortex_heart` (`漩涡之心`), `placeholder`。

### 11.1.1 土石类细分

土石类当前先细分到以下层级:

- `powder`: `sand` (`沙子`), `gravel` (`沙砾`), `soil` (`土壤`)
- `solid`: `sandstone` (`砂岩`), `raw_stone` (`原石`), `obsidian` (`黑曜石`)

当前先认为这一层已经足够支撑首版玩法。

### 11.2 植物类

植物类当前明确由以下 5 种不同材质构成:

- `root` (`树根`)
- `log` (`原木`)
- `leaf` (`树叶`)
- `vine` (`藤蔓`)
- `moss` (`苔藓`)

其中当前已经明确的行为有:

- `root` 会向下生长。
- `log` 会向上生长, 且随机变为 `leaf`。
- `vine` 和 `moss` 作为独立材质保留, 不再用 `菌丝 / 菌毯 / 苔藓` 这组重复结构。

### 11.3 金属类

金属类当前先只保留:

- `iron` (`铁`)
- `gold` (`金`)

### 11.3.1 常规水系

常规水系当前先只保留一个核心物质:

- `water` (`水`)

也就是说:

- 不额外拆出更多“常规液体”种类。
- 常规流体玩法当前先主要由 `水` 承担。

### 11.4 危险液体系

此前提到的 `树脂`, `毒泥`, `酸泥` 当前全部删除。

当前改为:

- `poison` (`毒液`)
- `acid` (`酸液`)
- `oil` (`油`)

这里的约束是:

- 它们在常温下表现为 `liquid`。
- 它们可以有低温冻结版本, 但冻结版本属于低温相变结果, 不是常温主形态。

### 11.5 特殊功能物质

#### 11.5.1 `explosive`

- `explosive` (`炸药`) 的核心特点是: 一旦燃烧, 会通过反应快速释放大量气体与热量。

#### 11.5.2 `phosphor_*`

- `荧光粉末` 不是单一物质, 而是按光类型分别存在。
- 当前统一采用:
  - `phosphor_visible`
  - `phosphor_holy`
  - `phosphor_chaos`
  - `phosphor_magic`
- 每一版 `phosphor_*` 都只发出自己对应的光。

#### 11.5.3 `pollution`

- `pollution` (`污染物`) 不受重力影响。
- 它的失重是物理性质, 不是反应。
- 它的扩散 / 转化通过与其他物质的配对反应实现。

#### 11.5.4 `vortex_heart`

- `vortex_heart` (`漩涡之心`) 会通过自身反应持续快速吸气 / 放气。
- 它通过这一行为持续制造风力。

#### 11.5.5 `fire`

- `fire` (`火焰`) 是正式物质, 不是纯温度状态。
- 绝大多数“燃烧”行为都应通过 `fire` 与其他物质的反应表达。

#### 11.5.6 `placeholder`

- 初代只保留一种统一的 `placeholder`, 而不是每种怪物一套 placeholder 物质。
- 除显式特例外, `placeholder` 按一般物质参与反应。

### 11.6 物质的相态原则

当前首版采用以下总原则:

- 每种物质族默认都应尽量具备完整的 `liquid / gas / powder / solid` 变种。
- 例外情况是: 如果该物质在高温或其他条件下会先被别的反应直接销毁或转化, 则不强求完整相变链。
- 对这种例外, 可以把相变温度设到极大值, 让它在正常玩法中不进入对应相态。

### 11.7 物质命名规则

当前统一采用以下命名约定:

- 基础物质名统一采用 `lower_snake_case`。
- 同一物质在不同相态下, 运行时名字统一写成 `material_phase`。
- `phase` 当前固定为 `solid / powder / liquid / gas`。
- 同一物质的 4 相不再各自起独立名字, 而统一使用同一个 `material` 前缀。

例如:

- `water_solid`
- `water_powder`
- `water_liquid`
- `water_gas`

其他物质后续也都按这一规则命名, 例如:

- `raw_stone_solid`
- `poison_liquid`
- `fire_gas`

光的运行时命名也统一采用 `lower_snake_case`, 当前固定为:

- `visible_light`
- `holy_light`
- `chaos_light`
- `magic_light`

这里的约束是:

- 这是运行时和数据表中的统一命名规则。
- 面向玩家显示时, 仍然可以有单独的中文显示名。

## 12. 首版光构成

当前已经明确的光包括:

- `visible_light` (`可见光`)
- `holy_light` (`圣光`)
- `chaos_light` (`混沌光`)
- `magic_light` (`魔法光`)

其中:

- `visible_light` 会升温, 且所有效果都乘 `dose`; 因此常规强度下很弱, 只有极强的光才会明显生效。
- `holy_light` 会对所有物质降温, 并增加 `integrity`; 对 `pollution` 的特例是 `convert_material = none`, 同时掉 `integrity`。
- `chaos_light` 会轻微升温, 并通过矩阵中逐项乱填的方式把物质随机 `convert_material` 成别的物质; 它不会再次放出自身。
- `magic_light` 用来挂载玩家自定义内容。

这里也延续前文约束:

- `Light` 流派允许使用现有光作为载体。
- 当前不允许 `LLM` 创造新的光定义。
- 因此 `Light` 流派的变化主要来自“现有光载体 + 不同反应”的组合。

## 13. 对偶反应表的语义

当前已经明确:

- 对两个对象之间的反应表, 语义应该是不对称的。
- 它描述的不是“两边同时发生什么”, 而是“如果邻居是什么, 则自己发生什么”。

也就是说:

- 一条配对反应规则的含义应理解为: `if neighbor matches X, self does Y`。
- 如果希望两个对象都发生变化, 应写成两条方向相反的规则, 而不是一条双向规则。

这一点会直接影响:

- 材材反应表。
- 材气反应表。
- 材光反应表。
- 其他所有带 lhs / rhs 的配对反应结构。

### 13.1 `convert_material` 的控制字段

当前已经明确:

- `convert_material` 不能只带目标物质 id。
- `convert_material` 还应带一个 `integrity_threshold`。
- `convert_material` 还应带一个每帧生效的 `harm_per_frame`。

其语义是:

- 反应命中后, 每帧只减少这条规则给出的 `harm_per_frame`。
- 只有当当前物质的 `integrity` 低于 `integrity_threshold` 时, 才真正执行 `convert_material`。

这样做的目的有 2 个:

- 用 `harm_per_frame + integrity_threshold` 控制反应速度。
- 避免为了“先掉血再转化”单独占用一个额外反应槽。

这里的 `harm_per_frame` 是 `convert_material` 项自己的属性, 不是独立反应类型。

### 13.2 当前已经确定的非空反应

以下只记录当前已经明确的非空反应, 其他默认先视为无反应。

#### 13.2.1 火焰类

- 燃烧统一做成 `fire` 与其他物质的反应。
- 同一套火焰反应至少包含两个温度区间:
  - 低温区间: 目标掉 `integrity`
  - 高温区间: 目标 `convert_material` 成 `fire`
- `oil` 和 `explosive` 都有自身反应: 温度达到阈值时会自己燃烧。
- `explosive` 在燃烧时, 无论是自发燃烧还是被点燃, 都会额外释放大量气体。
- 不存在“引信”这一额外概念。
- `fire` 自身有一个自身反应: 温度过低时 `convert_material = none`。
- `fire` 和各类液体之间不写直接反应, 统一靠温度系统处理。

#### 13.2.2 植物类

- 除 `vine` 外, 当前植物类已经确定的行为都可写入文档。
- `vine` 的反应暂时留空。
- 植物的向上 / 向下生长使用 `emit_material` 实现。
- `root` 通过 `emit_material` 向下生长。
- `log` 通过 `emit_material` 向上生长, 且随机变为 `leaf`。
- 所有植物类当前都可燃。

关于 `emit_material`, 当前还明确了一个执行语义:

- 如果发射出来的是 `powder`, 则 `direction` 描述的是速度方向 / 速度语义。
- 如果发射出来的是 `solid`, 则它没有飞行速度, 这时 `direction` 描述的是“放在当前格子的哪个位置”。

#### 13.2.3 酸类

- `acid` 会侵蚀普通土石, 植物等常规物质。
- `acid` 对 `obsidian`, `gold`, `pollution` 不生效。
- `acid` 的主要后果是让目标最终 `convert_material = none`。
- 不同物质在酸中掉 `integrity` 的速度不同。
- 不同物质触发 `convert_material = none` 的 `integrity_threshold` 不同。

#### 13.2.4 毒类

- `poison` 会使 `root / log / leaf / vine / moss` 掉 `integrity`。
- `poison` 会使 `placeholder` 掉 `integrity`。

#### 13.2.5 污染类

- `pollution` 的失重属于物理性质, 不属于反应。
- `pollution` 的转化效果只能通过配对反应表达。
- 也就是说, 相关反应应写成 `(other_material, pollution) -> other_material convert_material ...`, 而不是写成 `pollution` 自己主动改周围。

#### 13.2.6 发光类物质

- 发光类物质只负责发出各自对应的光。
- 当前最明确的这类物质是各个版本的 `phosphor_*`。

#### 13.2.7 Placeholder 特例

- `placeholder` 作为一般物质参与反应。
- 目前已经明确的特例包括:
  - 遇到 `poison` 时掉 `integrity`
  - 遇到 `chaos_light` 时掉 `integrity`
  - 遇到 `holy_light` 时增加 `integrity`

#### 13.2.8 光类反应

- `holy_light`:
  - 对所有物质降温
  - 对所有物质增加 `integrity`
  - 对 `pollution` 的特例是 `convert_material = none`, 同时掉 `integrity`
- `visible_light`:
  - 对所有物质升温
  - 所有效果都乘 `dose`
  - 因此普通强度下效果很小
- `chaos_light`:
  - 轻微升温
  - 通过在矩阵里逐项乱填的方式把物质随机 `convert_material` 成另一种物质
  - 当前不额外引入一个新的“random_convert_material 反应原语”

## 14. 气体模型方向

### 14.1 基本状态量

当前气体模型的方向明确为:

- 每种气体通道的 `c_i` 表示该气体的绝对浓度, 不是占比。
- \(\sum_i c_i\) 表示当前位置的总物质量。
- 世界中存在一个正式的 `空气` 通道, 作为背景空气载体。

### 14.1.1 首版气体 species 清单

当前首版气体 species 先收束为与以下几类相变或功能直接相关:

- `water` 的相变气体。
- `poison` 的相变气体。
- `oil` 的相变气体。
- `pollution` 的相变气体。

也就是说, 当前首版的独立气体通道重点只围绕:

- 常规水系。
- 危险液体系中的 `poison`。
- 油类。
- 污染物。

除此之外的其他气体 species 暂不优先展开。

### 14.2 压强与密度

当前采用的关系是:

\[
p = T \sum_i k_i c_i
\]

\[
\rho = \sum_i m_i c_i
\]

其中:

- \(T\) 是局部温度。
- \(k_i\) 是各气体对压强的系数。
- \(m_i\) 是各气体对密度的系数。

也就是说:

- 压强 `p` 来自所有气体通道的当前浓度与温度。
- 密度 `\rho` 也来自所有气体通道的当前浓度。

### 14.3 抽气 / 注气

当前已经明确:

- 抽气和注气都通过修改各气体通道的绝对浓度完成。
- 当某处被抽空时, 只允许把对应浓度减到 `0`, 不允许把浓度做成负值。

这样做的原因是:

- 负浓度会破坏物质语义。
- 负浓度会破坏后续的反应, 平流和扩散解释。
- 更强的吸力效果应来自更大的压强梯度, 而不是负浓度。

因此当前方向是:

- 抽气先把局部气体抽到接近真空或直接到 `0`。
- 若还需要更强的吸力, 则通过加大环境压强或压强梯度来增强效果。

### 14.4 风场更新

当前的模型方向是:

- 每帧先根据当前的 `c_i` 和 `T` 计算 `p` 与 `\rho`。
- 再结合上一帧的速度场 `u`, 去推进下一帧的风场。
- 得到的风场再反过来参与各气体通道的后续流动。

这里当前只定到原则层:

- 风场 `u` 和各气体浓度通道并存。
- 压强和密度由浓度与温度导出。
- 吸气 / 放气 / 风力物体都首先体现为对这套共享空气系统的作用。

`vortex_heart` 当前就是这一方向下的典型物质:

- 它持续快速吸气 / 放气。
- 它通过这一行为制造压强差和风力。

## 15. 当前已定与待定

### 15.1 当前已定

- 游戏是 `探索 + 战斗`。
- 玩家有 `HP` 和 `MP`。
- 实时施法只在当前装备的 `6` 个法术中选择。
- 施法时有 `bullet time`。
- 实时施法拆成 `法术选择`, `定位解析`, `强度估计` 三个模块。
- `法术选择` 是已装备法术上的 rerank 问题。
- `定位解析` 是固定语义空间中的定位提取问题。
- `强度估计` 优先依赖原始音频而非文本。
- `LLM 创造法术` 是非实时的高智力流程。
- `释放方式` 在法术创造阶段预先确定。
- `LLM` 创造能力当前分为 `反应`, `物质 / 气体`, `ForceSource`, `法术` 四层。
- 法术当前分为 `Gas / Material / Light / Force / Change` 五个流派。
- `Material / Gas` 流派的核心是“创建载体, 再由载体反应生效”。
- `Light` 流派允许基于现有光载体绑定不同反应, 但不允许创造新的光定义。
- `Change` 当前只允许修改区域内的 `物质类型 / 温度 / 速度`。
- 当前暂时不允许 `LLM` 创造新的光定义。
- 首版物质大类当前已确定到 `土石类 / 植物类 / 金属类 / 危险液体系 / 特殊功能物质` 这一层。
- 土石类当前细分为:
  `powder = sand / gravel / soil`,
  `solid = sandstone / raw_stone / obsidian`。
- 植物类当前确定包含 `root / log / leaf / vine / moss`。
- 金属类当前确定包含 `iron / gold`。
- 常规水系当前先只保留 `water`。
- 危险液体系当前确定包含 `poison / acid / oil`。
- 特殊功能物质当前确定包含 `fire / explosive / phosphor_* / pollution / vortex_heart / placeholder`。
- 同一物质四相的运行时命名当前统一采用 `material_phase` 规则。
- 首版光当前确定包含 `visible_light / holy_light / chaos_light / magic_light`。
- 对偶反应表当前按不对称语义理解: “若邻居为 X, 则自己执行 Y”。
- `convert_material` 当前应带 `target_material + integrity_threshold + harm_per_frame` 这组控制量。
- 气体模型当前按“绝对浓度 `c_i` + 温度导出 `p` 与 `\rho`”的方向理解。
- 首版气体 species 当前优先围绕 `water / poison / oil / pollution` 的相变或功能展开, 不再包含酸类通道。
- 抽气当前只允许把气体浓度减到 `0`, 不允许负浓度; 更强吸力靠更大的压强梯度。
